# BGK Todo UI 디자인 감사 보고서

> 작성 기준: UIdesigner.md 15년차 시니어 UI 디자이너 관점
> 평가 대상: BGK 팀 TODO 통합관리 시스템 (현재 버전)
> 분석 소스: `src/styles.ts`, `src/constants.ts`, 전체 컴포넌트 코드

---

## 총평

전체적으로 깔끔한 톤을 유지하고 있으나, **세부 수치의 일관성**과 **인터랙션 피드백**에서 완성도를 끌어올릴 여지가 많습니다.
특히 styles.ts에 **hover 상태와 transition이 전혀 정의되지 않은 점**이 가장 큰 갭입니다.

## 1. 타이포그래피

### [Critical] 반복 배지 폰트 9px — 가독성 저해

- **위치**: RepeatBadge 컴포넌트 (리스트/칸반/캘린더/메모 전체)
- **현재**: `fontSize: 9px`, `fontWeight: 700`
- **문제**: 9px은 대부분의 모니터에서 읽기 어려움. 특히 한글은 영문보다 획이 복잡해 작은 크기에서 뭉개짐
- **제안**: `10px` 이상으로 상향. 배지 내부 padding도 `2px 6px`으로 함께 조정
- **적용 범위**: RepeatBadge가 표시되는 리스트/칸반/캘린더/메모 뷰 전체

### [High] 헤더 버튼·배지 폰트 10px — 어두운 배경에서 가독성 부족

- **위치**: 헤더 영역 (프로젝트, 설정, 휴지통, 단축키 버튼 + 사용자 배지)
- **현재**: `fontSize: 10px`, 흰색 텍스트 on `#172f5a` 배경
- **문제**: 10px 흰색 텍스트는 어두운 배경에서도 세밀한 글자 형태를 잃기 쉬움. 버튼 라벨이 아이콘 보조 역할이라면 괜찮으나, 주요 기능 진입점이라면 부족
- **제안**: `11px`로 상향, 또는 아이콘 크기를 18px로 키워 텍스트 의존도를 줄임
- **적용 범위**: S.hBtn, S.hBdg

### [High] 한글 letter-spacing 0.5px — 테이블 헤더 가독성 저해

- **위치**: 테이블 헤더 (`S.th`)
- **현재**: `letterSpacing: '0.5px'`, `fontSize: 11px`, `fontWeight: 700`
- **문제**: letter-spacing은 영문 대문자 라벨에 적합한 기법. 한글은 자간이 벌어지면 오히려 글자 간 연결성이 끊겨 읽기 어려움
- **제안**: `letterSpacing: 0` 또는 `'0.2px'` 이하로 축소
- **적용 범위**: S.th (리스트 뷰 테이블 헤더)

### [Medium] line-height 미정의 — 한글 텍스트 답답함

- **위치**: 앱 전체 (`S.wrap`)
- **현재**: line-height 미지정 → 브라우저 기본값(약 1.2) 적용
- **문제**: 한글은 영문보다 글자 높이가 높아 1.2에서는 줄 간격이 답답하게 느껴짐
- **제안**: `S.wrap`에 `lineHeight: 1.5` 추가. 배지/버튼 등 한 줄 요소는 개별적으로 `lineHeight: 1` 지정
- **적용 범위**: 앱 전역

### [Medium] 폰트 굵기 500 vs 600 — 시각적 구분 불명확

- **위치**: 앱 전체
- **현재**: 비활성 탭/보조 버튼 = 500, 활성 탭/주요 버튼/배지 = 600
- **문제**: Pretendard에서 500과 600의 차이가 미미하여 굵기만으로 활성/비활성을 구분하기 어려움
- **제안**: 400(Regular) / 600(SemiBold)으로 간격을 넓혀 명확히 구분. 또는 색상 차이로 보완
- **적용 범위**: S.navB, S.bs, S.bp, S.badge

### [Low] 폰트 크기 5단계(9/10/11/12/13px) — 단계 과다

- **위치**: 앱 전체
- **현재**: 9, 10, 11, 12, 13px 총 5단계
- **문제**: 1px 차이는 사용자가 인식하기 어렵고, 디자인 시스템으로서 관리 복잡도 증가
- **제안**: 3~4단계로 축소 — `11px`(작은 라벨), `12px`(보조), `13px`(본문), `15px`(제목). 9px 제거, 10px은 11px로 통합
- **적용 범위**: 전체 S 객체

## 2. 색상 시스템

### [Critical] 테이블 헤더 색상 WCAG 대비율 미달

- **위치**: 리스트 뷰 테이블 헤더 (`S.th`)
- **현재**: 텍스트 `#94a3b8` on 배경 `#f8fafc` → 대비율 약 **2.9:1**
- **문제**: WCAG AA 기준 최소 4.5:1 필요. 시력이 좋지 않은 사용자는 헤더 텍스트를 읽기 어려움
- **제안**: 텍스트 색상을 `#64748b` (대비율 약 4.6:1) 이상으로 변경
- **적용 범위**: S.th

### [High] 테두리 색상 2개 혼용 — 의도 불명확

- **위치**: 앱 전체
- **현재**: 카드 테두리 `#e8edf4`, 내비/테이블/칩 테두리 `#e2e8f0`
- **문제**: 두 색상의 차이가 육안으로 거의 구분 불가(명도 차이 약 2%). 의도적 구분이 아니라면 관리 혼란
- **제안**: `#e2e8f0` 하나로 통일 (Tailwind slate-200 계열로 일관성 확보)
- **적용 범위**: S.card border, S.nav borderBottom, S.th boxShadow, S.pchip border

### [High] Primary 파란색(#2563eb) 과다 사용 — 의미 혼동

- **위치**: 앱 전체
- **현재**: 활성 탭, 포커스 링, 인라인 입력 테두리, 진행중 상태, 보통 우선순위가 모두 `#2563eb`
- **문제**: "진행중"과 "보통 우선순위"가 같은 파란색이면, 카드에 둘 다 표시될 때 의미 구분 불가
- **제안**: 보통 우선순위를 `#3b82f6`(밝은 파랑) 또는 `#6366f1`(인디고) 계열로 분리하여 Primary와 구분
- **적용 범위**: INIT_PRI_C['보통'], INIT_PRI_BG['보통']

### [Medium] 위험 계열 색상 중복

- **위치**: 앱 전체
- **현재**: `#dc2626`(긴급 우선순위, 위험 버튼 텍스트)과 `#ef4444`(프로젝트 팔레트)가 혼용
- **문제**: 같은 빨간 계열인데 미묘하게 다른 값 사용. 프로젝트 색상이 "긴급"으로 오인될 수 있음
- **제안**: 위험/긴급 전용 빨강은 `#dc2626`으로 통일, 프로젝트 팔레트에서 `#ef4444`를 `#f43f5e`(로즈) 또는 다른 계열로 교체
- **적용 범위**: PROJ_PALETTE, INIT_PRI_C

### [Medium] 프로젝트 팔레트 보라색 3개 — 구분 어려움

- **위치**: 사이드바, 칸반 필터 칩, 캘린더 이벤트 칩
- **현재**: `#8b5cf6`, `#6366f1`, `#a855f7` — 세 가지 보라/인디고 계열이 팔레트에 동시 존재
- **문제**: 프로젝트가 3개 이상일 때 보라 계열 프로젝트끼리 색상 구분이 거의 불가
- **제안**: 보라 계열을 2개 이하로 줄이고, 대신 `#0d9488`(틸), `#78716c`(갈색) 등 다른 색역 추가
- **적용 범위**: PROJ_PALETTE

### [Low] 아바타 색상과 프로젝트 색상 겹침

- **위치**: 리스트/칸반/대시보드에서 아바타와 프로젝트 도트가 동시 표시되는 곳
- **현재**: AVATAR_COLORS에 `#2563eb`, `#d97706` 등 — PROJ_PALETTE와 겹치는 색상 존재
- **문제**: 같은 행에서 담당자 아바타와 프로젝트 도트가 같은 파란색이면 혼동
- **제안**: 아바타 색상을 프로젝트 팔레트와 겹치지 않는 톤(더 어둡거나 더 밝은)으로 분리
- **적용 범위**: AVATAR_COLORS

## 3. 간격·여백

### [High] 4px 그리드 미준수 — 전체 리듬감 불일치

- **위치**: 앱 전체 (S 객체)
- **현재**: 2, 3, 5, 6, 9, 10, 14px 등 4의 배수가 아닌 값 다수
- **문제**: 간격 체계가 불규칙하면 요소 간 정렬이 미세하게 어긋나 "왜인지 모르게 지저분해 보이는" 느낌 발생
- **제안**: 4px 기반 그리드로 정리

| 현재 | 제안 | 대상 |
|------|------|------|
| 2px | 4px | 내비 탭 gap |
| 3px | 4px | 배지 padding-top |
| 5px (radius) | 6px | 인라인 입력 border-radius |
| 6px | 8px | 주요 버튼 padding-top |
| 9px | 8px | 배지 padding-left |
| 10px | 12px | 테이블 셀 padding, 내비 탭 padding-top |
| 14px | 12px 또는 16px | 주요 버튼 padding-left |

- **적용 범위**: S 객체 전체

### [Medium] 카드 padding 16px vs 모달 padding 20px — 불일치

- **위치**: S.card, S.mbox
- **현재**: 카드 `padding: 16px`, 모달 `padding: 20px`
- **문제**: 같은 "흰색 컨테이너" 역할인데 내부 여백이 다름. 모달 안에 카드가 들어가면 이중 여백 차이가 드러남
- **제안**: 둘 다 `20px`로 통일하거나, 카드 `16px` / 모달 `24px`로 명확한 차등
- **적용 범위**: S.card, S.mbox

### [Medium] 헤더(54px) + 내비(40px) = 94px 상단 고정 영역

- **위치**: 앱 레이아웃 상단
- **현재**: 총 94px이 viewport 상단에 고정
- **문제**: 1080p 기준 약 9%, 768p(태블릿) 기준 약 12% 차지. 콘텐츠 영역이 좁아질 수 있음
- **제안**: 헤더를 48px, 내비를 36px로 줄이면 총 84px (10px 절약). 또는 스크롤 시 내비만 숨기기
- **적용 범위**: S.hdr height, S.nav height, S.nav top, 메인 콘텐츠 offset

## 4. Hover·인터랙션 효과

### [Critical] styles.ts에 hover 상태·transition 전혀 미정의

- **위치**: `src/styles.ts` 전체
- **현재**: hover 색상, transition 속성이 S 객체에 하나도 없음. 각 컴포넌트가 `onMouseEnter/Leave`로 개별 처리
- **문제**:
  - 같은 버튼이 A 컴포넌트에서는 hover 시 어두워지고 B에서는 안 변하는 등 불일치 발생 가능
  - transition 없이 색상이 즉시 바뀌면 딱딱하고 저품질 느낌
  - hover 상태를 추가/수정할 때 모든 컴포넌트를 찾아다녀야 함
- **제안**: S 객체에 다음 항목 추가

```typescript
// hover 상태 색상
hoverBg: { primary: '#1d4ed8', secondary: '#e2e8f0', danger: '#fee2e2' },

// 기본 transition (모든 인터랙티브 요소에 적용)
transition: { fast: 'all 0.12s ease', normal: 'all 0.2s ease', slow: 'all 0.3s ease' },

// active (클릭 순간) 효과
active: { transform: 'scale(0.97)' },
```

- **적용 범위**: bp, bs, bd, hBtn, navB, 카드, 테이블 행, 아이콘 버튼 등 모든 인터랙티브 요소

### [High] 주요 버튼(bp) hover 피드백 부재

- **위치**: "저장", "등록", "행 추가" 등 Primary 버튼
- **현재**: 그라데이션 `linear-gradient(135deg, #2563eb, #1d4ed8)` + `cursor: pointer`만 있음
- **문제**: hover 시 아무 변화 없으면 "이게 진짜 버튼인가?" 의문. 특히 disabled 상태와 시각적 차이 부족
- **제안**: hover 시 `filter: brightness(1.08)` + `boxShadow: '0 2px 8px rgba(37,99,235,.3)'` 추가
- **적용 범위**: S.bp 사용하는 모든 버튼

### [High] 카드 hover 효과 부재

- **위치**: 메모 뷰 카드, 칸반 뷰 카드
- **현재**: 카드 그림자 `0 1px 3px rgba(0,0,0,.07)` 고정
- **문제**: hover 시 그림자나 위치 변화가 없으면 카드가 클릭 가능하다는 인지가 부족
- **제안**: hover 시 `boxShadow: '0 4px 12px rgba(0,0,0,.12)'` + `transform: 'translateY(-2px)'` + `transition: all 0.2s ease`
- **적용 범위**: S.card

### [Medium] 내비 탭 hover 피드백 부재

- **위치**: 대시보드/리스트/캘린더/칸반 탭
- **현재**: 비활성 탭 `color: #64748b`, `background: none` — hover 시 변화 없음
- **문제**: 탭 위에 마우스를 올려도 클릭 가능한지 알기 어려움
- **제안**: hover 시 `background: '#f1f5f9'`, `color: '#334155'` + transition 0.15s
- **적용 범위**: S.navB (비활성 상태)

### [Medium] 테이블 행 hover — 현재 컴포넌트별 개별 처리

- **위치**: 리스트 뷰 테이블
- **현재**: ListView 내 `onMouseEnter`로 배경색 변경 — S 객체에 공통 정의 없음
- **문제**: hover 색상/transition이 컴포넌트에 하드코딩되어 일괄 변경 어려움
- **제안**: `S.rowHover = { background: '#f8fafc', transition: 'background 0.12s ease' }` 추가
- **적용 범위**: ListView, AddTodoSection 테이블 행

### [Low] active(누르는 순간) 효과 미적용

- **위치**: 모든 버튼
- **현재**: 클릭 시 시각적 반응 없음
- **문제**: 버튼을 눌렀는지 안 눌렀는지 촉각적 피드백 부재
- **제안**: 모든 버튼에 `:active` 또는 `onMouseDown` 시 `transform: scale(0.97)` 적용
- **적용 범위**: bp, bs, bd, hBtn, navB, 아이콘 버튼

## 5. 그림자·깊이

### [High] 그림자 Level 2(팝오버/드롭다운용) 부재

- **위치**: DropPanel, DateTimePicker, 각종 팝오버
- **현재**: Level 1(카드 — 거의 안 보임)과 Level 4(모달 — 매우 강함) 사이에 중간 단계 없음
- **문제**: 드롭다운/팝오버가 배경 위에 떠 있는 느낌이 불충분하거나, 컴포넌트마다 제각각 그림자 지정
- **제안**: S 객체에 그림자 단계 체계 추가

```typescript
shadow: {
  sm: '0 1px 3px rgba(0,0,0,.07)',          // Level 1: 카드
  md: '0 4px 12px rgba(0,0,0,.1)',           // Level 2: 드롭다운, 팝오버
  lg: '0 8px 20px rgba(0,0,0,.12)',          // Level 3: 플로팅 패널
  xl: '0 12px 28px rgba(0,0,0,.15)',         // Level 4: 모달
}
```

- **적용 범위**: DropPanel, DateTimePicker, 캘린더 팝오버, 토스트

### [Medium] 헤더 그림자 opacity 0.3 — 과도하게 무거움

- **위치**: 헤더 (`S.hdr`)
- **현재**: `boxShadow: '0 2px 8px rgba(23,47,90,.3)'`
- **문제**: 스크롤 시 콘텐츠 위에 진한 그림자가 겹쳐 시각적으로 무거움. 헤더가 "떠 있는" 느낌이 과도
- **제안**: `'0 1px 4px rgba(23,47,90,.12)'`로 약화 — 구분은 되지만 가볍게
- **적용 범위**: S.hdr

### [Low] 카드 그림자가 거의 안 보임

- **위치**: S.card
- **현재**: `0 1px 3px rgba(0,0,0,.07)` — 배경 `#f0f4f8`와 카드 `#fff`의 색상 차이 + 테두리가 이미 구분 역할
- **문제**: 그림자가 사실상 없는 것과 같음. 그림자를 쓸 거면 보이게, 안 쓸 거면 테두리만으로 통일
- **제안**: `'0 1px 4px rgba(0,0,0,.1)'`로 살짝 강화하거나, 그림자 제거 후 테두리만으로 정리
- **적용 범위**: S.card

---

## 6. 둥글기 (Border Radius)

### [High] 카드 10px vs 모달 16px — 같은 역할인데 다른 둥글기

- **위치**: S.card (`borderRadius: 10`), S.mbox (`borderRadius: 16`)
- **현재**: 둘 다 "흰색 박스 컨테이너"인데 둥글기가 다름
- **문제**: 사용자는 의식적으로 인지하지 못하지만, 무의식적으로 "뭔가 통일감이 없다"고 느낌
- **제안**: 둘 다 `12px`로 통일. 또는 카드 `8px`(작은 요소) / 모달 `12px`(큰 컨테이너)로 명확한 차등
- **적용 범위**: S.card, S.mbox

### [Low] 인라인 입력 border-radius 5px — 홀수

- **위치**: S.iinp
- **현재**: `borderRadius: 5` — 다른 모든 요소가 짝수(6, 8, 10, 14, 16)
- **문제**: 시스템 전체에서 유일한 홀수 값. 미세하지만 서브픽셀 렌더링에서 약간의 흐림 가능
- **제안**: `6px`로 변경
- **적용 범위**: S.iinp

---

## 7. 아이콘·시각 요소

### [Medium] 프로젝트 색상 도트 6×6px — 너무 작음

- **위치**: 리스트 뷰 행, 사이드바 필터, 칸반 카드 등 (`S.dot`)
- **현재**: `width: 6, height: 6, borderRadius: '50%'`
- **문제**: 6px 도트는 특히 밝은 색상(노란색, 연두색)에서 거의 안 보임. 프로젝트 구분의 핵심 시각 요소인데 역할을 못함
- **제안**: `8px`로 확대. 필요 시 `10px`까지. 도트 옆 텍스트와의 간격도 `4px → 6px` 확보
- **적용 범위**: S.dot, 사이드바/리스트/칸반/캘린더에서 도트를 사용하는 모든 곳

### [Medium] 아이콘 크기 불일치

- **위치**: 앱 전체
- **현재**: 컴포넌트마다 Heroicons를 `14px`, `15px`, `16px`, `18px` 등 다양한 크기로 사용
- **문제**: 같은 맥락(예: 사이드바 필터 라벨)에서 아이콘 크기가 다르면 정렬이 어긋남
- **제안**: 크기 단계를 3개로 정리 — `14px`(배지/칩 내), `16px`(기본), `20px`(강조/헤더)
- **적용 범위**: 아이콘을 사용하는 모든 컴포넌트

### [Low] 아이콘-텍스트 수직 정렬

- **위치**: 버튼, 배지, 필터 항목 등
- **현재**: `display: flex, alignItems: center`로 대부분 처리하지만, 일부 인라인 아이콘이 텍스트와 1~2px 어긋남
- **문제**: 아이콘이 텍스트 baseline과 미세하게 안 맞으면 "대충 만든" 느낌
- **제안**: 모든 아이콘+텍스트 조합에 `display: inline-flex, alignItems: center, gap: 4px` 패턴 통일
- **적용 범위**: 아이콘과 텍스트가 함께 쓰이는 모든 곳

---

## 8. 모달·오버레이

### [Medium] 모달 오버레이 opacity 0.5 — 과도하게 어두움

- **위치**: S.mOverlay
- **현재**: `background: 'rgba(0,0,0,.5)'`, `backdropFilter: 'blur(4px)'`
- **문제**: 0.5 + blur(4px)가 겹치면 뒤 화면이 거의 안 보임. 모달이 "무거운" 느낌
- **제안**: `rgba(0,0,0,.35)` + `blur(2px)`로 약화. 뒤 콘텐츠가 살짝 비치면서도 모달에 집중
- **적용 범위**: S.mOverlay (모든 모달에 적용)

### [Medium] 모달 등장 애니메이션 미확인

- **위치**: Modal 컴포넌트
- **현재**: 코드에서 애니메이션 관련 속성 없음 — 즉시 나타남
- **문제**: 갑자기 튀어나오는 모달은 사용자를 놀라게 함. 0.2초 fade-in이면 충분히 부드러움
- **제안**: 오버레이에 `opacity 0→1 (0.2s)`, 모달 박스에 `opacity 0→1 + translateY(8px→0) (0.25s)` CSS 애니메이션 추가
- **적용 범위**: Modal 컴포넌트

### [Low] 모달 max-width 580px — 편집 폼에서 좁을 수 있음

- **위치**: S.mbox
- **현재**: `maxWidth: 580`
- **문제**: EditForm의 2열 그리드(담당자/마감기한, 우선순위/상태) + 리치 에디터가 580px 안에서 답답할 수 있음
- **제안**: `640px`로 확대 검토. 또는 내용에 따라 모달 크기를 `sm(480px) / md(640px) / lg(800px)` 3단계로 분리
- **적용 범위**: S.mbox

---

## 9. 토스트·알림

### [Medium] 토스트 퇴장 애니메이션 부재

- **위치**: Toast 컴포넌트
- **현재**: 등장 시 slide-up 있으나, 시간 만료 시 즉시 사라짐
- **문제**: 갑자기 사라지면 "방금 뭔가 있었는데?" 느낌. 읽는 도중 사라질 수 있음
- **제안**: 퇴장 시 `opacity 1→0 + translateY(0→8px)` 0.3초 fade-out 추가
- **적용 범위**: Toast 컴포넌트

### [Low] 토스트 타입이 2개뿐 (성공/오류)

- **위치**: Toast 컴포넌트
- **현재**: `type: 'success' | 'error'` — 초록/빨강만 존재
- **문제**: 정보성 알림(파랑, "필터가 적용되었습니다")이나 경고(노랑, "네트워크 느림")에 적합한 타입 없음
- **제안**: `info`(파랑 `#2563eb`), `warning`(노랑 `#d97706`) 타입 추가
- **적용 범위**: Toast 컴포넌트, 타입 정의

## 10. 뷰별 시각 디테일

### [High] 리스트 뷰 — 선택 행 inset shadow 부자연스러움

- **위치**: ListView 선택된 행
- **현재**: 파란색 `inset box-shadow`로 테두리 효과 표현
- **문제**: inset shadow는 일반적인 "선택됨" 패턴이 아님. 사용자가 이것이 선택 상태인지 오류인지 혼동 가능
- **제안**: `background: '#eff6ff'` (밝은 파랑 배경) + 좌측 `3px solid #2563eb` 바로 교체. 더 표준적이고 깔끔
- **적용 범위**: ListView 행 선택 스타일

### [Medium] 메모 뷰 — 완료 카드 opacity 60% 과도

- **위치**: MemoView 완료 카드
- **현재**: `opacity: 0.6`, 회색 배경, 그림자 제거
- **문제**: 60%면 텍스트가 너무 흐려서 "무엇이 완료됐는지" 확인하기 어려움
- **제안**: `opacity: 0.75` + `filter: saturate(0.3)`로 색상만 빠지게. 텍스트는 읽을 수 있도록
- **적용 범위**: MemoView 완료 카드

### [Medium] 칸반 뷰 — 컬럼 배경과 카드 배경 구분 부족

- **위치**: KanbanView 컬럼
- **현재**: 컬럼 배경이 앱 배경(`#f0f4f8`)과 유사하고, 카드 배경이 `#fff`
- **문제**: 컬럼 영역의 경계가 불명확해서 4개 컬럼이 하나로 이어져 보일 수 있음
- **제안**: 컬럼 배경을 `#f8fafc`로 하고 `border: 1px solid #e2e8f0` + `borderRadius: 8px` 적용
- **적용 범위**: KanbanView 컬럼 컨테이너

### [Low] 캘린더 뷰 — 반복 업무 점선 vs 일반 실선 구분 미약

- **위치**: CalendarView 이벤트 칩
- **현재**: 반복 = 점선 테두리, 일반 = 실선 테두리
- **문제**: 작은 칩에서 점선/실선 차이가 눈에 잘 안 들어옴
- **제안**: 반복 업무에 ArrowPathIcon 아이콘을 칩 좌측에 작게(10px) 추가하여 이중 구분
- **적용 범위**: CalendarView 이벤트 칩

### [Low] 대시보드 — KPI 카드 hover 피드백 부재

- **위치**: Dashboard KPI 카드 4개
- **현재**: 클릭 시 리스트 뷰로 이동하지만 hover 시 커서/배경 변화 없음
- **문제**: 클릭 가능하다는 시각적 단서가 없어 사용자가 기능을 발견하지 못함
- **제안**: hover 시 `boxShadow` 강화 + `transform: translateY(-2px)` + `cursor: pointer` 추가
- **적용 범위**: Dashboard KPI 카드

# 기능 개발 워크플로우 (feature)

> 새 기능 요청 시 기획 → 데모 → 구현 → QA까지 단계별로 진행한다.

## 발동 조건

- 사용자가 `/feature` 또는 "새 기능 만들어", "기능 추가해" 요청 시
- 명확한 기능 요구사항이 있을 때

## 실행 절차

### Phase 1: 기획 (planning.md 활용)

1. 문제 정의 — 누가, 언제, 왜 이 기능이 필요한가?
2. A안 vs B안 비교 — 최소 2개 접근법 제시
3. MVP 범위 정의 — 1차에 꼭 필요한 것만
4. 데이터 구조 설계 — `types.ts`에 추가할 타입
5. 리스크 분석 — UX/기술/비즈니스

**사용자에게 기획안을 보여주고 방향 확정 후 다음 단계 진행**

### Phase 2: 데모 페이지

1. `demo/YYYYMMDD-기능명-demo.html` 생성
2. 앱 동일 스타일 (Pretendard, 앱 색상 팔레트)
3. 인터랙티브 프로토타입 (JS로 동작하는 데모)
4. Before/After 비교 (기존 vs 개선)
5. 구현 우선순위 표

**사용자에게 데모 확인 후 구현 착수**

### Phase 3: 구현

순서:
1. `types.ts` — 타입 정의
2. `useTodoApp.ts` — 상태 + CRUD + Firestore 연동
3. 컴포넌트 UI — 기존 공통 컴포넌트 재사용 (code.rule.md #2)
4. props 전달 — App.tsx → ListView → 하위 컴포넌트
5. `npx tsc --noEmit` — 타입 체크

### Phase 4: QA (QA.md 활용)

1. 기본 동작 — 생성/수정/삭제/조회
2. 설정 연동 — sync-check.md 실행
3. 크로스 뷰 — 모든 뷰에서 데이터 일치
4. 엣지 케이스 — 빈값, 최대값, 동시 접속
5. Firestore — 저장/복원 누락 없는지

### Phase 5: 배포

deploy.md 실행

## 주의사항

- Phase 1 → Phase 2는 사용자 확인 후 진행
- 기존 컴포넌트를 최대한 재사용 (새로 만들지 않음)
- 같은 기능이 여러 뷰에 영향을 주면 모든 뷰를 함께 수정

# 기획 리뷰 — 현재 앱 평가 및 개선 로드맵

## 이 프롬프트가 하는 일

현재 구현된 앱을 기획자 시각으로 평가하고, **무엇이 잘 되어 있는지 / 무엇이 부족한지 / 어떤 순서로 개선해야 하는지**를 정리합니다.

## 언제 이 프롬프트를 쓰는가

- **"지금 앱 어떤 것 같아?"** — 객관적인 기획 평가 요청 시
- **"다음에 뭘 만들면 좋을까?"** — 개선 우선순위 도출 시
- **새 기능 개발 전** — 기존 기능과 중복·충돌 여부 점검 시
- **정기 리뷰** — 분기별 앱 방향성 점검 시

## 이 프롬프트가 하지 않는 것

- 코드 직접 수정 (→ 구현은 직접 수행)
- 시각 디자인 디테일 (→ UIdesigner.md 사용)
- 버그 테스트 (→ QA.md 사용)

## 평가 프레임워크

### 1. 핵심 가치 명확성
> "이 앱이 없으면 팀이 어떤 불편을 겪는가?"

- 앱의 핵심 가치가 1문장으로 정리되는가?
- 사용자가 매일 쓰는 기능 Top 3는 무엇인가?
- 쓰지 않는 기능이 오히려 UI를 복잡하게 만들고 있지 않은가?

### 2. 기능 완성도 평가

| 기능 | 완성도 | 우선 개선 포인트 |
|------|--------|----------------|
| 업무 CRUD | — | — |
| 필터/검색 | — | — |
| 리스트 뷰 | — | — |
| 칸반 뷰 | — | — |
| 캘린더 뷰 | — | — |
| 대시보드 | — | — |
| AI 자동 입력 | — | — |
| 템플릿 | — | — |
| 팀/권한 관리 | — | — |
| 반복 업무 | — | — |

### 3. 사용성 갭 분석
> "기획 의도 vs 실제 사용 방식의 차이"

- 의도한 대로 쓰이고 있는 기능은?
- 의도와 다르게 쓰이거나 안 쓰이는 기능은?
- 사용자가 자주 우회하는 흐름은?

### 4. 중복·충돌 점검

- 같은 역할을 하는 기능이 두 곳 이상 존재하는가?
- 메뉴 구조에서 서로 혼동을 주는 항목이 있는가?
- 뷰 간 동일 데이터 표현 방식이 일관적인가?

### 5. 개선 우선순위 매트릭스

| 개선 항목 | 임팩트 | 난이도 | 우선순위 |
|----------|--------|--------|---------|
| — | 높음/중/낮음 | 높음/중/낮음 | P1/P2/P3 |

**우선순위 기준:**
- **P1**: 임팩트 높음 + 난이도 낮음 → 즉시 실행
- **P2**: 임팩트 높음 + 난이도 높음 → 다음 스프린트
- **P3**: 임팩트 낮음 → 백로그

## 개선 로드맵 작성 기준

### Phase 1 — 빠른 개선 (1~2주)
기존 기능의 UX 개선, 버그 수정, 누락된 피드백 추가

### Phase 2 — 기능 강화 (1개월)
사용 빈도 높은 기능의 완성도 향상, 새 기능 MVP 추가

### Phase 3 — 확장 (분기)
새 뷰, 통합 기능, 성능 최적화, 모바일 대응
